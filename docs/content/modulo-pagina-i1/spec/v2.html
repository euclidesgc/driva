Módulo Página (I1) — Construtor de Páginas SDUI

Aguardando aprovação do board  Spec v2   2026-06-22 · Issue BMJ-2 · 1º incremento

v2 — o que mudou. Esta versão corrige o layout do editor: de 4 colunas (herança da POC WidgetMill) para 3 painéis (painel esquerdo paleta/tree · canvas central phone mockup · inspector direito), alinhado ao web-prototipe oficial e ao modelo FlutterFlow que inspirou o Driva. O restante do conteúdo é igual à v1. Ver v1 (superada).
Esta spec detalha apenas o incremento I1: montar uma página SDUI arrastando widgets para um canvas central, organizá-los pela tree view, editar propriedades e ver o preview fiel ponta a ponta — salvando como rascunho. Filtros/condições (I2), widget avançado (I3) e workflow/versionamento (I4) ficam fora. Contexto: Visão Geral · PRD: PRD v2.

1. Objetivo

Entregar o núcleo do Page Builder: um editor onde um usuário compõe uma página a partir de um catálogo de widgets, organiza-os em ordem, ajusta suas propriedades e vê em tempo real, num preview 100% fiel ao Flutter, como a página aparecerá no app — com molduras de dispositivo para testar responsividade. Ao final, a página é salva como rascunho e pode ser reaberta.

★ Requisito central (decisão do board). O maior problema da POC foi o design ruim. Portanto, em I1 a qualidade de UX e visual é critério de aceite, não um extra: editor limpo, moderno, intuitivo e agradável; design system consistente (referência: web-prototipe); interações fluidas (arrastar, selecionar, editar); hierarquia visual clara. Uma página funcional porém feia/confusa não cumpre o I1.

2. Escopo

Dentro do I1

  • Editor com 3 painéis (seção 5), reusando o preview fiel da POC.
  • Catálogo inicial: ~14 primitivos Flutter (Decisão I1.1 resolvida).
  • Arrastar widgets da paleta para o canvas; reordenar via tree view; selecionar e remover.
  • Inspector: editar as propriedades do widget selecionado (forms gerados pelo schema Zod).
  • Preview fiel (Flutter Web em iframe via postMessage) + seletor de dispositivos (smartphone/tablet/desktop).
  • Salvar/reabrir a página como rascunho, escopada por project_id.
  • Design system + UX de alto padrão — referência visual: docs/web-prototipe/.

Fora do I1

  • Condições de exibição + filtros por widget + simulação de usuário no preview → I2.
  • Construtor de widget avançado (estados + fonte de dados) → I3.
  • Workflow rascunho→homologação→produção, papéis, versionamento, agendamento → I4.
  • Serving para o app de produção (GET /pages) — em I1 o preview interno basta.

3. Conceitos

ConceitoDefinição (I1)
PáginaFragmento SDUI: uma lista ordenada de widgets (blocos) que o app renderiza em uma tela já existente (Decisão I1.3 resolvida).
Bloco (widget na página)Instância de um widget do catálogo com propriedades próprias; referencia o widget por slug + version.
CatálogoConjunto de widgets disponíveis para arrastar — ~14 primitivos Flutter (Decisão I1.1).
Rascunho (draft)Cópia de trabalho da página, salva e reabrível. Sem máquina de estados (isso é I4).
Preview fielRenderização pelo mesmo renderer do app (sdui_flutter), embarcado via iframe em postMessage.

4. Modelo da página (spec JSON)

A página é dado puro — o mesmo espírito do kernel da POC. Esboço (modelagem fina = responsabilidade do Architect):

{
  "specVersion": 1,
  "kind": "page_fragment",
  "id": "home-banner-section",
  "name": "Seção Banner — Home",
  "project_id": "proj-abc",
  "blocks": [
    { "id": "b1", "widget": { "slug": "text", "version": 1 }, "props": { "value": "Promoção!" } },
    { "id": "b2", "widget": { "slug": "image", "version": 1 }, "props": { "src": "{{banner_url}}" } }
  ]
  // "visibility" por bloco e condições da página entram no I2
}

Cada bloco referencia o widget por slug + version (imutável). A modelagem detalhada de IDs e referências é responsabilidade do Architect no plano de fases.

5. Anatomia do editor — 3 painéis CORRIGIDO em v2

Layout de 3 painéis, alinhado ao web-prototipe oficial e ao FlutterFlow. A coluna "Organização" separada da POC foi eliminada: a ordem dos blocos é controlada diretamente no canvas (arrastar) e pela Tree View no painel esquerdo.

③ PAINEL DIREITO

Inspector: sem seleção → Page Props (nome, project_id). Com bloco selecionado → formulário Zod do widget. Alterações refletem no canvas com debounce ≤ 300 ms.

Topbar: logo Driva · breadcrumb (App / NomeDaPágina) · barra de busca · botões (Preview, Publish). Em I1 o botão Publish aparece desabilitado com tooltip "disponível no I4". Indicador de status "Salvo / Editando…".

Painéis redimensionáveis via drag nos divisores esquerdo/canvas e canvas/direito. Undo/redo (Ctrl+Z/Y) para todas as ações de edição.

6. Preview fiel (herdado da POC)

Mesma arquitetura validada no WidgetMill: app Flutter Web (sdui_preview) embarca o renderer real (sdui_flutter), servido como asset estático na mesma origem do editor; o editor envia o spec da página por postMessage (com validação de origin) e o Flutter re-renderiza. O renderer passa a entender, além de widgets isolados, uma página = lista de blocos. Fidelidade 100%, zero divergência React↔Flutter.

7. Requisitos não-funcionais

AspectoCritério em I1
UX / visualCritério de aceite. Fidelidade ao web-prototipe: design moderno, limpo, intuitivo; design system consistente; interações fluidas.
PerformancePreview carrega < 3 s; re-renderiza com debounce sem janking. Build Flutter cacheado.
SegurançaValidação de origin no postMessage. Auth JWT. Multi-tenant via project_id.
AcessibilidadeNavegação por teclado, contraste WCAG AA.
ExtensibilidadeCatálogo extensível adicionando entradas ao schema sem alterar código do editor.

8. Decisões do I1

✓ Todas resolvidas pelo board em 2026-06-21/22:
  1. Paleta do I1: os ~14 primitivos do Flutter da POC (Container, Row, Column, Text, Image, Icon, Button, Spacer, Divider, Card, Stack, Padding, Expanded, SizedBox) como blocos. Seções de alto nível ficam para I3+.
  2. Design antes de codar: sim — o web-prototipe é a referência visual primária; o Architect define o design system na 1ª fase do plano.
  3. Endereçamento da página: páginas são fragmentos plugados em telas existentes do app (não rotas de tela inteira). Modelo: page_fragment (ver seção 4).
  4. Layout do editor: 3 painéis — painel esquerdo (paleta/tree toggle) · canvas central (phone mockup Flutter) · painel direito (inspector). Eliminada a coluna "Organização" separada da POC.

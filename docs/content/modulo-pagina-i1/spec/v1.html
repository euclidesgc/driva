Módulo Página (I1) — Construtor de Páginas SDUI

Suspenso — aguardando design system   Versão de trabalho · 2026-06-21 · Issue BMJ-2 · 1º incremento

⏸ Atividades suspensas (decisão do board, 2026-06-21). O board está construindo o design system no Claude Design e pediu para suspender todo o trabalho até avisar o CEO onde o design estará. Como UX/design é o requisito central do I1 (e o board optou por “UX antes de codar”), faz sentido aguardar o design system antes de detalhar wireframes, plano e implementação. Retomamos quando o board indicar a localização do design.
Esta spec detalha apenas o incremento I1: montar uma página SDUI arrastando widgets, ordená-los, editar propriedades e ver o preview fiel ponta a ponta — salvando como rascunho. Filtros/condições de exibição (I2), construtor de widget avançado (I3) e fluxo/versionamento/agendamento (I4) ficam fora deste incremento. Contexto: Visão Geral.

1. Objetivo

Entregar o núcleo do Page Builder: um editor onde um usuário compõe uma página a partir de um catálogo de widgets, organiza-os em ordem, ajusta suas propriedades e vê em tempo real, num preview 100% fiel ao Flutter, como a página aparecerá no app — com molduras de dispositivo para testar responsividade. Ao final, a página é salva como rascunho e pode ser reaberta.

★ Requisito central (decisão do board). O maior problema da POC foi o design ruim. Portanto, em I1 a qualidade de UX e visual é critério de aceite, não um extra: editor limpo, moderno, intuitivo e agradável; design system consistente; interações fluidas (arrastar, selecionar, editar); hierarquia visual clara. Uma página funcional porém feia/confusa não cumpre o I1.

2. Escopo

Dentro do I1

  • Editor com layout em colunas (seção 4), reusando o preview fiel da POC.
  • Catálogo inicial de widgets na paleta (ver Decisão I1.1).
  • Arrastar widgets da paleta para a página; reordenar, selecionar e remover.
  • Inspector: editar as propriedades do widget selecionado (forms gerados por tipo, herdados do kernel).
  • Preview fiel (Flutter Web em iframe via postMessage) + seletor de dispositivos (iPhone/Android/tablet).
  • Salvar/reabrir a página como rascunho (working copy), escopada por project_id.
  • Design system + UX de alto padrão (requisito central).

Fora do I1 (backlog dos próximos incrementos)

  • Condições de exibição da página e filtros por widget + simulação de propriedades de usuário no preview → I2.
  • Construtor de widget avançado (estados loading/success/error/idle/empty + fonte de dados) → I3.
  • Workflow rascunho→homologação→produção, papéis por etapa, versionamento e agendamento → I4.
  • Entrega/serving de páginas para o app de produção (GET /pages) — em I1 o preview basta.

3. Conceitos

ConceitoDefinição (I1)
PáginaArtefato SDUI: uma lista ordenada de widgets (blocos) que o app renderiza. Tem identidade (ver Decisão I1.3).
Bloco (widget na página)Instância de um widget do catálogo posicionada na página, com suas próprias props. Referencia o widget do catálogo.
CatálogoConjunto de widgets disponíveis para arrastar (ver Decisão I1.1).
Rascunho (draft)Cópia de trabalho da página, salva e reabrível. Sem máquina de estados ainda (isso é I4).
Preview fielRenderização da página pelo mesmo renderer do app (sdui_flutter), embarcado via iframe.

4. Modelo da página (spec JSON)

A página é dado, no mesmo espírito do kernel da POC. Esboço (a fechar no plano técnico):

{
  "specVersion": 1,
  "kind": "page",
  "slug": "home",
  "name": "Home",
  "blocks": [
    { "id": "b1", "widget": { "slug": "banner", "version": 2 }, "props": { "title": "Promo", "image": "{{...}}" } },
    { "id": "b2", "widget": { "slug": "product_list", "version": 1 }, "props": { "source": "..." } }
  ]
  // "visibility" por bloco e condições da página entram no I2
}

Cada bloco referencia um widget do catálogo por slug + version (imutável), garantindo que a página renderize de forma estável. A modelagem fina (inline vs. referência, IDs) é detalhe do plano do Architect.

5. Anatomia do editor (layout em colunas)

Como descrito pelo board, com UX caprichada:

1 · PALETAWidgets disponíveis, arrastáveis para a página. Busca e agrupamento por categoria.
2 · ORGANIZAÇÃOOs blocos da página, em ordem. Arrastar para reordenar, selecionar, remover. (Filtro por bloco = I2.)
3 · PROPRIEDADESInspector do bloco selecionado: forms gerados por tipo de widget.
4 · PREVIEWRender fiel no app, com moldura de dispositivo (iPhone/Android/tablet). (Simular usuário = I2.)

Topbar: nome da página, salvar rascunho, indicador de “salvo”. Painéis redimensionáveis. O preview atualiza com debounce a cada alteração. Interações de arrastar/soltar precisam ser visivelmente fluidas e previsíveis (zona de drop clara, feedback de seleção, undo/redo).

6. Preview fiel (herdado da POC)

Mesma arquitetura validada na POC: app Flutter Web (sdui_preview) embarca o renderer real (sdui_flutter), servido como asset estático na mesma origem do editor; o editor envia o spec da página por postMessage (com validação de origin) e o Flutter re-renderiza. O renderer passa a entender, além de widgets isolados, uma página = lista de blocos. Fidelidade 100%, zero divergência React↔Flutter.

8. Decisões do I1

✓ Resolvidas pelo board em 2026-06-21:
  1. Paleta do I1: os ~14 primitivos do Flutter da POC (Container, Row, Column, Text, Image…) como blocos. Seções de alto nível ficam para depois.
  2. UX antes de codar: sim — wireframes/protótipo + design system precedem a implementação.
  3. Endereçamento da página: páginas são fragmentos anexados a telas que já existem no app (não rotas/slugs de tela inteira). O modelo da página (seção 4) permanece uma lista ordenada de blocos, mas a unidade é um fragmento plugado numa tela existente — a fechar no plano técnico.

Pendência externa: o board está produzindo o design system no Claude Design e suspendeu o trabalho até indicar onde ele estará.

Registro original das perguntas (histórico):

  1. O que popula a paleta de widgets no I1? Como o construtor de widget avançado (I3) ainda não existe, o I1 precisa de um catálogo inicial.

    ▸ Recomendado: Conjunto curado de “seções” de alto nível (ex.: Banner, Carrossel, Lista de itens, Card, Texto, Botão, Imagem, Espaçador) — mais próximo de páginas SDUI reais e de uma UX atrativa.
    ○ Os ~14 primitivos do Flutter da POC (Container, Row, Column, Text, Image…) como blocos.
    ○ Híbrido: algumas seções de alto nível + alguns primitivos.

  2. Investir em UX/design antes de codar (dado que foi o problema nº1 da POC)?

    ▸ Recomendado: Sim — UX entrega wireframes/protótipo do editor e definimos um design system antes de implementar, para garantir o padrão visual exigido.
    ○ Não — ir direto à implementação, melhorando a UI sobre a base da POC.

  3. Como a página é endereçada/consumida pelo app?

    ▸ Recomendado: Cada página tem um slug/rota (ex.: /home, /promo) que o app pede ao SDUI.
    ○ Páginas são fragmentos/feeds anexados a telas já existentes do app.
    ○ Ambos (página completa e fragmento).

Plano de Implementação — Módulo Página (I1)

Aguardando aprovação do board  Plano v2   2026-06-23 · Issue BMJ-3 · Architect

v2 — o que mudou vs v1: (1) Aninhamento em I1: Fase 2 atualizada para modelo em árvore; Fase 6 com DnD de 3 cenários; Fase 7 com tree hierárquica real. (2) F3 e F4 paralelas: execução simultânea reduz o timeline. (3) 7 decisões técnicas incorporadas: Zustand, @dnd-kit, auth via env, um draft por projeto, auto-save opcional, inspector tipos expandidos, navegação simplificada. Ver v1 (superada).
Documentos-base: Spec v3 · PRD v2 · docs/WidgetMill/ (referência técnica) · docs/web-prototipe/ (referência visual primária).
Critério de aceite de 1ª classe (do board). Uma página funcional com visual/UX ruim não passa no I1. O design system e a fidelidade ao web-prototipe são validados em cada fase — não só no final.

Visão Geral do Plano

O I1 é construído em 8 fases, com as Fases 3 e 4 rodando em paralelo. A sequência garante: (a) fundações sólidas antes da UI; (b) modelo de dados em árvore desde a Fase 2 — sem refatoração posterior; (c) preview Flutter validado cedo (maior risco técnico); (d) design system guiando o desenvolvimento desde a Fase 1.

2
Kernel de Página & Schemas Zod — Modelo em Árvore atualizado v2
Depende de: Fase 1

Objetivo: packages/spec do Driva com modelo de dados em árvore (PageFragment + Block recursivo com child/children) e os 14 schemas Zod completos — fonte única de verdade para editor, backend e Flutter.

  • todo Definir schema Zod Block recursivo: { id: string, widget: { slug: string, version: number }, props: Record<string, unknown> } onde props pode conter child?: Block (single-child) ou children?: Block[] (multi-child). Usar z.lazy() para recursão.
  • todo Definir schema Zod PageFragment: { specVersion: 1, kind: "page_fragment", id: string, name: string, project_id: string, blocks: Block[] }. O array blocks contém os blocos raiz; filhos vivem em props.child/props.children.
  • todo Portar e adaptar os 14 schemas de widgets do WidgetMill (docs/WidgetMill/packages/spec/src/nodes/): Folha: text, image, icon, spacer, divider, button. Single-child: container, padding, expanded, sizedBox, card. Multi-child: row, column, stack. Auditar cada schema; completar props faltantes para I1.
  • todo Criar catálogo de widgets (widgetCatalog): { slug, version, displayName, icon, category, slotKind: "leaf"|"child"|"children", schema, defaultProps }. Categorias: Layout / Conteúdo / Interação.
  • todo Utilitários de kernel: generateBlockId(), createBlock(slug, version), validatePage(json), walkBlocks(page, visitor) (traversal recursivo).
  • todo Utilitários de árvore: insertBlock(page, parentId, slotKey, index, block), removeBlock(page, blockId), moveBlock(page, blockId, newParentId, newIndex). Essas operações são usadas pelo Zustand store no editor. Implementar aqui no kernel (puro, sem dependência de UI) para testabilidade.
  • todo Exportar tipos TypeScript inferidos do Zod: PageFragment, Block, WidgetDef, SlotKind.
  • todo Testes unitários do kernel: validação de PageFragment com blocos aninhados; insertBlock, removeBlock, moveBlock com árvores de profundidade 3; catálogo com 14 entradas.

Critério de saída: pnpm -F @driva/spec test passa; schema valida árvore com Row → Padding → Text; utilitários de árvore corretos.

3
Backend NestJS — API de Rascunhos ▶ paralela com F4
Depende de: Fase 2

Objetivo: API NestJS para salvar e carregar rascunhos de página, com JWT via env, isolamento multi-tenant por project_id e persistência em Postgres via Prisma.

  • todo Scaffold apps/api NestJS: módulos AppModule, PagesModule, AuthModule.
  • todo Schema Prisma: modelo Page (id, projectId, name, draft JSONB, createdAt, updatedAt). Migration inicial. Segurança Validar draft com schema Zod antes de persistir — nunca armazenar JSON arbitrário bruto.
  • todo Endpoints REST: GET /pages?projectId= · GET /pages/:id · POST /pages · PUT /pages/:id · DELETE /pages/:id.
  • todo Auth JWT: middleware valida Bearer token e injeta projectId. Segredo em JWT_SECRET env; expiração configurável. Segurança Em I1: um token por projeto (I4 traz roles). Sem tela de login.
  • todo Guard multi-tenant: toda query filtra por projectId do JWT — nunca do body. Segurança
  • todo Validação de payload: DTO (class-validator) + schema Zod do kernel como segunda barreira.
  • todo CORS restrito (ALLOWED_ORIGIN env) + Helmet. Segurança
  • todo Testes de integração (Supertest): CRUD completo; isolamento de tenants; JWT inválido → 401. Segurança

Critério de saída: CRUD funciona via curl; testes de integração passam; tenant A não acessa dados do B.

Flutter Preview — Adaptação para Páginas em Árvore ▶ paralela com F3 atualizado v2
Depende de: Fase 2

Objetivo: Portar o renderer Flutter da POC para renderizar uma PageFragment em árvore. O renderer já suporta child/children — a adaptação é de entry point e protocolo postMessage.

  • todo Portar sdui_flutter do WidgetMill para flutter/sdui_flutter do Driva. Adaptar entry point para aceitar PageFragment (lista de blocos raiz, cada um com subárvore em props.child/props.children).
  • todo Implementar PageRenderer Flutter: widget que recebe PageFragment e renderiza os blocos raiz numa Column scrollável. A renderização de cada bloco é recursiva (já implementada no renderer da POC para o modelo child/children). Bloco com widget desconhecido → placeholder visual com slug e label.
  • todo Portar sdui_preview para o Driva. Protocolo postMessage: aceitar { type: "update_page", payload: PageFragment }; responder { type: "render_complete" }. Segurança Validar event.origin contra allowlist configurada em build time. Rejeitar silenciosamente origens desconhecidas.
  • todo Build Flutter Web do sdui_preview → apps/web/public/preview/. Script automatizado na raiz. Asset gitignored; documentado no CLAUDE.md.
  • todo Teste manual: página mínima com Column → Padding → Text enviada via postMessage; confirmar renderização correta.
  • todo Testes Flutter: PageFragment com árvore aninhada (3 níveis); widget desconhecido mostra placeholder; PageFragment vazia → sem crash.

Critério de saída: iframe renderiza árvore de 3 níveis via postMessage; origin inválida rejeitada.

6
Paleta & Drag-and-Drop — 3 cenários de drop atualizado v2
Depende de: Fase 5

Objetivo: Painel esquerdo em modo Paleta e DnD para inserir widgets no canvas com suporte a 3 cenários: (a) inserção na raiz da página, (b) inserção como filho de container, (c) inserção entre irmãos.

  • todo Painel esquerdo — Modo Paleta: grid de 14 widgets (ícone + nome), agrupados por categoria (Layout / Conteúdo / Interação). Campo de busca por nome (filtro client-side).
  • todo DnD com @dnd-kit/core: cada widget na paleta é Draggable. Ghost preview durante arrasto (miniatura com nome do widget).
  • todo Cenário A — drop na raiz: drop na área do canvas fora de qualquer bloco → insere no final de page.blocks. Indicador visual: linha de inserção ao final da lista.
  • todo Cenário B — drop dentro de container: drag sobre um bloco container (Row, Column, Stack, Card, Padding, Container, Expanded, SizedBox) → highlight do container como drop zone; soltar insere como filho. Distinguir single-child vs multi-child via slotKind do catálogo. O canvas Flutter é um iframe opaco — o drop zone é um overlay React sobre o canvas. A posição relativa do arrasto determina o alvo por consulta ao estado da árvore (mapa de blocos para posições de tela enviados via postMessage pelo renderer).
  • todo Cenário C — drop entre irmãos: linha de inserção aparece entre blocos irmãos ao arrastar sobre um container multi-child. Soltar insere na posição indicada.
  • todo Ao soltar: chamar insertBlock(page, parentId, slotKey, index, newBlock) do kernel, selecionar o bloco inserido, disparar update do preview via usePreviewBridge (debounce 300ms).
  • todo Protocolo de layout do Flutter → editor: sdui_preview envia { type: "layout_update", blockRects: { [blockId]: DOMRect } } após cada render, para o overlay calcular drop zones.
  • todo Undo/redo: ação de drop salva snapshot no histórico Zustand. Ctrl+Z reverte; Ctrl+Y reaplicar.
  • todo Acessibilidade: widgets na paleta acessíveis por teclado (Tab + Enter insere ao final da raiz como alternativa ao arrastar).

Critério de saída: PM arrasta Text para o canvas → aparece. Arrasta Row, depois arrasta Text dentro do Row → preview mostra texto dentro de Row. Undo reverte os dois drops.

7
Tree View Hierárquica, Seleção & Inspector atualizado v2
Depende de: Fase 6

Objetivo: Toggle Paleta/Tree, tree view genuinamente hierárquica com drag-and-drop de reordenação/reparenting, seleção sincronizada canvas↔tree, e inspector com formulários Zod gerados automaticamente.

  • todo Toggle "Paleta / Tree" no cabeçalho do painel esquerdo.
  • todo Tree View hierárquica: renderização recursiva dos blocos da página. Cada nó mostra ícone (do catálogo), slug do widget e id curto. Containers exibem seus filhos indentados e colapsáveis.
  • todo DnD sortable na Tree View com @dnd-kit/sortable: reordenar irmãos e reparentar (mover nó para dentro de outro container). Ao soltar, chamar moveBlock(page, blockId, newParentId, newIndex) do kernel.
  • todo Seleção sincronizada: clicar em nó da tree → seleciona bloco (highlight na tree + outline no canvas via postMessage). Clicar no canvas → protocolo de seleção reversa: Estratégia: o overlay React sobre o canvas recebe o clique; calcula qual bloco foi tocado usando o mapa de blockRects (recebido do Flutter na Fase 6). Seleciona o bloco mais interno que contém o ponto clicado. Se nenhum → deseleciona.
  • todo Highlight de seleção no canvas: sdui_preview recebe { type: "select_block", blockId } e renderiza outline de seleção + label sobre o bloco selecionado.
  • todo Remoção de bloco: botão remover visível no hover de cada nó da tree. Ação chama removeBlock(page, blockId) (remove o bloco e toda a subárvore). Undo restaura.
  • todo Painel direito — Page Props (sem seleção): campo editável para name da página.
  • todo Painel direito — Widget Inspector (com seleção): formulário gerado pelo schema Zod do widget selecionado. Tipos suportados em I1: string (input texto), number (input numérico), boolean (toggle), enum (select/radio), Color (color picker com input hex), EdgeInsets (4 campos numéricos: top/right/bottom/left). Props do tipo child/children são gerenciadas pelo DnD — não aparecem no inspector de I1.
  • todo Debounce 300ms: cada alteração no inspector atualiza page.blocks e dispara usePreviewBridge.
  • todo Undo/redo completo: reordenar na tree, reparentar, remover bloco (e subárvore), editar propriedade — todos registrados no histórico Zustand. Ctrl+Z / Ctrl+Y.
  • todo Acessibilidade do inspector: Tab entre campos, labels associados a inputs, contraste WCAG AA.

Critério de saída: Usuário monta Column → Padding → Text; reordena e reparenta pela tree; edita props; preview atualiza em <300ms; undo/redo cobre todas as ações.

8
Persistência, Polimento UX & Homologação
Depende de: Fases 3, 7

Objetivo: Integrar save/load com o backend, polimento UX final, Security Gate 3 e homologação do board contra o DoD do I1.

  • todo Integração API: "Salvar rascunho" → PUT /pages/:id. Abrir editor → GET /pages/:id (um draft por project_id). Status indicator atualizado após cada operação.
  • todo Auto-save (opcional / Should): debounce 2s após última edição → salva silenciosamente. Indicador discreta de "Salvando…".
  • todo Polimento: transições suaves nos toggles, hover states em todos os elementos interativos, ghost preview fluido no DnD, toast de confirmação ao salvar/remover.
  • todo Performance: medir carregamento inicial do iframe Flutter (<3s); validar debounce 300ms sem janking; cachear build Flutter (Service Worker ou HTTP headers).
  • todo Auditoria de acessibilidade: navegação por teclado no inspector e paleta; contraste WCAG AA (axe-core / Lighthouse).
  • todo Security Gate 3: revisão de código: (1) origin validation no postMessage; (2) JWT secret em env, não em código; (3) tenant isolation — toda query filtra por projectId do JWT; (4) CORS restrito; (5) XSS — inspector React sanitiza strings em labels/tooltips; (6) payload size limits na API (prevenir DoS via JSON gigante). Segurança
  • todo Testes E2E (Playwright): fluxo completo — abrir editor → arrastar 3 widgets → aninhar em Column → editar props → salvar → reabrir → rascunho restaurado.
  • todo Sessão de homologação com o board: validar os 8 itens do DoD (PRD seção 7), com atenção especial ao item 7 (UX agradável e intuitivo).
  • todo Atualizar docs/manifest.json com status aprovado. Fechar issues BMJ-3 e criar issues de implementação por fase.

Critério de saída: Todos os 8 itens do DoD verificados; Security Gate 3 concluído; testes E2E passam; I1 homologado pelo board.

Decisões técnicas de projeto

AspectoDecisãoRazão
Modelo de dadosÁrvore recursiva — child/children em propsConsistente com WidgetMill POC; renderer Flutter já implementado; todos os 14 widgets úteis desde I1.
Estado globalZustand + immer + middleware de históricoMenos boilerplate; undo/redo nativo; immer facilita mutações imutáveis na árvore.
Drag-and-drop@dnd-kit/core + sortableAcessível; funciona com iframe como alvo; sortable para tree view.
Drop zones no canvasOverlay React + mapa de blockRects via postMessageIframe Flutter é opaco para o DOM React. O renderer envia bounding boxes após cada render; o overlay usa para calcular targets.
Inspector formsGerador customizado a partir do schema ZodControle total sobre o visual do inspector; sem dependência de biblioteca de forms pesada.
Auth em I1JWT via env, sem login screenI4 traz roles e login. I1 precisa apenas de isolamento por projeto — token configurado externamente.
Navegação de páginasUm draft por project_id, editor diretoSimplifica I1. Page list e criação de múltiplas páginas entram em I2+.
Auto-saveOpcional (Should, não Must)RF-12 exige save explícito. Auto-save é conforto adicional — não bloqueia homologação.

Riscos e mitigações

RiscoProb.Mitigação
Drop zones em canvas iframe — cálculo de blockRects imprecisoAltaProtótipo técnico no início da F6. Fallback: drop always ao final da raiz + reordenar via tree.
Performance da árvore recursiva no Zustand (atualizações frequentes)Médiaimmer torna as atualizações de árvore O(profundidade). Memoização por blockId com useSelector. Debounce 300ms evita re-renders excessivos.
UX/visual reprovado pelo boardMédiaValidação visual ao final das F1, F5 e F7 — não acumular dívida.
Performance do iframe Flutter (TTI > 3s)MédiaBuild pré-compilado; medir na F4; Service Worker para cache.
Reparenting na tree view ambíguo (onde inserir)MédiaIndicadores visuais claros (linha entre irmãos vs highlight de container). UX inspirado no FlutterFlow.

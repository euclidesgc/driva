Plano de Implementação — Módulo Página (I1)

Aguardando aprovação do board  Plano v1   2026-06-23 · Issue BMJ-3 · Architect

Documentos-base: Spec v2 (aguardando aprovação) · PRD v2 (aguardando aprovação) · docs/WidgetMill/ (POC de referência técnica) · docs/web-prototipe/ (referência visual primária).
Critério de aceite de 1ª classe (do board). Uma página funcional com visual/UX ruim não passa no I1. O design system e a fidelidade ao web-prototipe são validados em cada fase — não só no final.

Visão Geral do Plano

O I1 é construído em 8 fases sequenciais, cada uma entregando um incremento funcional verificável. A sequência foi desenhada para: (a) estabelecer fundações sólidas antes de construir UI; (b) validar o preview Flutter cedo, pois é o componente de maior risco técnico; (c) garantir que o design system guia o desenvolvimento desde o início, evitando retrabalho de visual na fase final.

F1 Infra & Monorepo → F2 Kernel & Schemas → F3 Backend NestJS → F4 Flutter Preview → F5 Editor Shell & Canvas → F6 Paleta & DnD → F7 Tree, Seleção & Inspector → F8 Persistência, Polimento & Homologação

Fases

1
Infra, Monorepo & Design System
Depende de: — (ponto de partida)

Objetivo: Ter o repositório Driva estruturado como monorepo pnpm (espelho do WidgetMill), o ambiente de desenvolvimento funcionando, e o design system extraído do web-prototipe documentado como tokens prontos para uso no editor React.

  • todo Criar estrutura monorepo pnpm na raiz do repo (pnpm-workspace.yaml, package.json raiz, .nvmrc, .editorconfig). Espelhar layout do WidgetMill: apps/web, packages/spec, flutter/sdui_flutter, flutter/sdui_preview, infra/.
  • todo Configurar ferramentas de qualidade: ESLint (config compartilhada), TypeScript (strict), Prettier, .gitignore atualizado.
  • todo Infraestrutura local: docker-compose.yml com Postgres 16 + Redis (para sessões/cache futuro). Arquivo .env.example documentado. Segurança Nunca commitar .env real. Variáveis sensíveis apenas em runtime.
  • todo Extrair design tokens do web-prototipe em arquivo packages/design-tokens/tokens.ts (ou CSS variables): Cores: fundo canvas #eef0f2, painéis brancos, accent #6c4cf1, texto, muted, bordas. Tipografia: Public Sans (Google Fonts ou self-hosted). Espaçamento, border-radius, sombras, estados de interação.
  • todo Criar apps/web Next.js 15 boilerplate (App Router) com design tokens aplicados globalmente. Página de demonstração dos tokens (cores, tipografia, espaçamento) para validação visual do board.
  • todo Atualizar CLAUDE.md na raiz com instruções do monorepo Driva (comandos, convenções, estrutura).

Critério de saída: pnpm install funciona na raiz; docker-compose up sobe Postgres; página de tokens renderiza no browser com visual idêntico ao web-prototipe.

2
Kernel de Página & Schemas Zod
Depende de: Fase 1

Objetivo: Criar o packages/spec do Driva com o modelo de dados da página (page_fragment) e os schemas Zod completos dos ~14 widgets primitivos — a fonte única de verdade usada pelo editor React, backend NestJS e renderer Flutter.

  • todo Definir e implementar o schema Zod PageFragment: specVersion: number, kind: "page_fragment", id: string, name: string, project_id: string, blocks: Block[]
  • todo Definir schema Block: id: string (nanoid), widget: { slug: string, version: number }, props: Record<string, unknown>. IDs de bloco são gerados no cliente (nanoid 8 chars). Imutáveis após criação em I1.
  • todo Auditar e portar os schemas dos ~14 primitivos Flutter do WidgetMill (docs/WidgetMill/packages/spec): Container, Row, Column, Text, Image, Icon, Button, Spacer, Divider, Card, Stack, Padding, Expanded, SizedBox. Para cada widget: validar que o schema Zod cobre todas as props necessárias para I1. Completar gaps detectados.
  • todo Criar catálogo de widgets (widgetCatalog): array com { slug, version, displayName, icon, category, schema, defaultProps }. Categorias iniciais: Layout (Container, Row, Column, Stack, Padding, Expanded, SizedBox, Spacer, Divider) · Conteúdo (Text, Image, Icon) · Interação (Button) · Agrupamento (Card).
  • todo Escrever utilitários de kernel: generateBlockId(), createBlock(slug, version) (retorna Block com defaultProps), validatePage(json).
  • todo Exportar tipos TypeScript inferidos do Zod (PageFragment, Block, WidgetDef) como API pública do pacote.
  • todo Testes unitários do kernel: validação de PageFragment válida e inválida; criação de Block com defaultProps; catálogo completo com 14 entradas.

Critério de saída: pnpm -F @driva/spec test passa; tipos exportados funcionam no editor e no backend.

3
Backend NestJS — API de Rascunhos
Depende de: Fase 2

Objetivo: API NestJS funcional para salvar e carregar rascunhos de página, com autenticação JWT, isolamento multi-tenant por project_id e persistência em Postgres via Prisma.

  • todo Scaffold apps/api com NestJS CLI: módulos AppModule, PagesModule, AuthModule.
  • todo Schema Prisma: modelo Page (id, projectId, name, draft JSONB, createdAt, updatedAt). Migration inicial. Segurança O campo draft armazena o JSON da página — validar com schema Zod antes de persistir (nunca armazenar JSON arbitrário bruto).
  • todo Endpoints REST: GET /pages?projectId= — listar rascunhos do projeto.
    GET /pages/:id — carregar rascunho por ID.
    POST /pages — criar novo rascunho.
    PUT /pages/:id — atualizar rascunho (upsert).
    DELETE /pages/:id — remover rascunho.
  • todo Autenticação JWT: middleware que valida Bearer token e injeta projectId no request. Segurança Tokens assinados com segredo em variável de ambiente. Expiração configurável. Em I1: um token por projeto (granularidade fina é I4).
  • todo Guard de multi-tenant: toda query filtra por projectId extraído do JWT — nunca aceitar projectId do body. Segurança
  • todo Validação de payload: DTO com class-validator que rejeita JSON inválido antes de chamar Prisma. Schema Zod do kernel invocado na camada de serviço como segunda barreira.
  • todo Testes de integração (Supertest): CRUD completo; isolamento de tenants (project A não acessa dados do project B); JWT inválido retorna 401. Segurança
  • todo CORS: permitir apenas origem do editor (ALLOWED_ORIGIN via env). Headers de segurança básicos (Helmet). Segurança

Critério de saída: CRUD de páginas funciona via curl/Insomnia; testes de integração passam; tenant A não consegue ler dados do tenant B.

4
Flutter Preview — Adaptação para Páginas
Depende de: Fase 2

Objetivo: Adaptar o renderer Flutter da POC (sdui_flutter) para renderizar uma page_fragment (lista de blocos) em vez de widget isolado, e atualizar o sdui_preview (app iframe) para receber e exibir páginas via postMessage.

  • todo Portar sdui_flutter do WidgetMill para flutter/sdui_flutter no repo Driva. Adaptar o entry point para aceitar PageFragment (lista de blocos) além do widget isolado da POC.
  • todo Implementar PageRenderer Flutter: widget que recebe PageFragment e renderiza os blocos em sequência (Column scrollável por padrão em I1). Blocos com widget desconhecido renderizam um placeholder visual com label do slug.
  • todo Portar sdui_preview para o Driva. Atualizar protocolo postMessage: aceitar mensagens do tipo { type: "update_page", payload: PageFragment }. Segurança Validar event.origin contra allowlist de origens configurada em build time. Rejeitar silenciosamente mensagens de origem desconhecida.
  • todo Implementar protocolo de confirmação: após receber e renderizar, sdui_preview envia { type: "ready" | "render_complete" } ao editor.
  • todo Build Flutter Web do sdui_preview → apps/web/public/preview/. Script/task automatizado na raiz do monorepo. Asset gitignored; instrução de build documentada no CLAUDE.md.
  • todo Teste manual: editor mínimo (HTML puro ou Storybook) que envia uma PageFragment de exemplo ao iframe e confirma renderização correta dos ~14 widgets.
  • todo Testes Flutter (flutter test): renderização de PageFragment com 1+ blocos; widget desconhecido mostra placeholder; PageFragment vazia renderiza tela vazia (sem crash).

Critério de saída: iframe recebe uma PageFragment via postMessage e renderiza todos os 14 widgets; origin inválida é rejeitada.

5
Editor React — Shell, Canvas & Topbar
Depende de: Fases 1, 2, 4

Objetivo: Montar o shell completo do editor (3 painéis redimensionáveis, topbar, canvas com phone mockup) com fidelidade visual ao web-prototipe e o iframe Flutter integrado, mas ainda sem lógica de drag-and-drop ou inspector.

  • todo Layout de 3 painéis com CSS Grid/Flexbox: painel esquerdo (largura padrão 220px), canvas central (flex), painel direito (220px). Divisores arrastáveis (resize handles) nos separadores esquerdo/canvas e canvas/direito.
  • todo Topbar: logo Driva, breadcrumb ("App / NomeDaPágina"), busca global (placeholder em I1), botões (código, Preview, Publish desabilitado com tooltip "disponível no I4"). Indicador de status ("Salvo" / "Editando…") à direita.
  • todo Canvas central: fundo #eef0f2, moldura de celular (phone mockup com bordas arredondadas e notch) centralizada. iframe /preview/index.html dentro da moldura. Toolbar do canvas: zoom (+/−, % exibida, padrão 90%), seletor de device (Smartphone 393×852, Tablet 768×1024, Desktop 1280×800), dimensões em px.
  • todo Integração postMessage: hook React (usePreviewBridge) que envia PageFragment ao iframe com debounce 300ms e escuta confirmações. Segurança Enviar mensagens apenas para a origem do iframe; nunca expor segredos no postMessage payload.
  • todo Trocar device: redimensiona a moldura via CSS transform/scale sem recarregar o iframe (contentWindow.postMessage de resize, ou simples resize da div envoltória).
  • todo Componentes de design system reutilizáveis: Button, Tooltip, Input, Badge, Divider — com tokens da Fase 1. Documentados em Storybook (opcional, se não travar o progresso).
  • todo Estado global do editor com Zustand (ou Context+Reducer): { page: PageFragment, selectedBlockId: string|null, isDirty: boolean }. Zustand preferido: menos boilerplate, suporte nativo a undo/redo via middleware immer+history.
  • todo Validação visual: board revisa screenshot/demo do shell antes de avançar para DnD. Garantir fidelidade ao web-prototipe.

Critério de saída: Editor abre no browser; 3 painéis redimensionáveis; phone mockup com iframe Flutter visível; troca de device funciona sem reload; visual fiel ao web-prototipe.

6
Paleta & Drag-and-Drop (Paleta → Canvas)
Depende de: Fase 5

Objetivo: Implementar o modo Paleta no painel esquerdo e a interação de arrastar um widget da paleta para o canvas (phone mockup), inserindo o bloco na página com valores padrão e selecionando-o automaticamente.

  • todo Avaliar e escolher biblioteca DnD: @dnd-kit/core (recomendado — acessível, sem dependência HTML5 nativa, melhor suporte a drop em iframe) vs HTML5 DnD API. Decisão: @dnd-kit, pois o drop target final é um iframe Flutter — exige coordenadas calculadas, não evento de drop nativo dentro do iframe.
  • todo Painel esquerdo — Modo Paleta: grid de widgets (~14 itens), ícone SVG + nome, agrupados por categoria (Layout / Conteúdo / Interação / Agrupamento). Campo de busca por nome (filtro client-side).
  • todo Draggable: cada widget na paleta é um DraggableItem; ao iniciar arrasto, exibe ghost preview (miniatura com nome).
  • todo Drop zone no canvas: área interna da moldura do celular é um Droppable. Durante arrasto, exibe highlight azul na borda da moldura + linha de inserção indicando posição.
  • todo Ao soltar no canvas: criar novo Block via createBlock(slug, version) do kernel, inserir no array page.blocks na posição correta, selecionar o bloco recém-criado. Preview Flutter atualiza via postMessage com debounce 300ms.
  • todo Undo/redo: ação de drop registrada no histórico. Ctrl+Z remove o bloco inserido; Ctrl+Y o re-insere.
  • todo Acessibilidade: widgets da paleta acessíveis por teclado (tab + enter para inserir ao final da página como alternativa ao arrastar).

Critério de saída: PM arrasta qualquer dos 14 widgets e o vê aparecer no preview Flutter; undo remove; busca filtra a paleta.

7
Tree View, Seleção no Canvas & Inspector
Depende de: Fase 6

Objetivo: Completar a interatividade do editor — toggle Paleta/Tree, seleção de blocos (canvas e tree), reordenação via tree, remoção, e o painel Inspector com formulários gerados pelo schema Zod.

  • todo Toggle "Paleta / Tree" no cabeçalho do painel esquerdo. Estado local no componente do painel.
  • todo Tree View: lista hierárquica dos blocos da página com ícone, nome do widget e label do bloco. Em I1, hierarquia é plana (lista) — aninhamento real fica para quando os widgets containerizados (Row, Column, Stack) forem suportados com filhos. Registrar como limitação conhecida.
  • todo Drag-and-drop dentro da Tree View para reordenar blocos. Usar @dnd-kit/sortable. Atualiza page.blocks e dispara re-render do preview.
  • todo Seleção sincronizada: clicar em nó da tree seleciona o bloco (destaca no canvas com outline); clicar no canvas seleciona o bloco (ativa o nó na tree). O canvas Flutter não pode ouvir eventos DOM diretamente — a seleção via clique no canvas exige overlay React transparente posicionado sobre o iframe, que detecta o clique e calcula o bloco em foco por posição. Ou: usar mensagens postMessage do preview indicando qual bloco foi tocado.
  • todo Remoção: botão remover em cada nó da tree (visível no hover). Confirmação visual (toast ou undo imediato sem diálogo modal). Undo restaura o bloco.
  • todo Painel direito — Page Props (sem seleção): campos editáveis para name da página.
  • todo Painel direito — Widget Inspector (com seleção): gerador automático de formulário a partir do schema Zod do widget selecionado. Tipos suportados em I1: string, number, boolean, enum. Tipos complexos (listas, objetos aninhados) ficam para I2/I3.
  • todo Debounce 300ms: cada alteração no inspector atualiza page.blocks[i].props e dispara usePreviewBridge com o JSON atualizado.
  • todo Undo/redo para: reordenar na tree, remover bloco, editar propriedade no inspector. Histórico de até 50 ações.
  • todo Acessibilidade do inspector: navegação por teclado (Tab entre campos), labels associados, contraste WCAG AA.

Critério de saída: Usuário monta página com 3+ widgets, reordena via tree, edita props e vê preview atualizar em <300ms; undo/redo funciona para todas as ações.

8
Persistência, Polimento UX & Homologação
Depende de: Fases 3, 7

Objetivo: Integrar save/load de rascunho com o backend, aplicar o polimento UX final (animações, estados, micro-interações), realizar a revisão de segurança (Security Gate 3) e conduzir a homologação do board contra o DoD do I1.

  • todo Integração API: botão "Salvar rascunho" → PUT /pages/:id. Carregar rascunho ao abrir editor → GET /pages/:id. Indicador de status ("Salvo" / "Editando…") atualizado após cada operação.
  • todo Auto-save opcional: debounce de 2s após última edição → salva silenciosamente. Respeitar RF-12 (salvar/carregar rascunho).
  • todo Polimento de interações: transições suaves nos toggles de painel, hover states em todos os elementos interativos, ghost preview fluido no DnD, toast de confirmação ao salvar/remover.
  • todo Revisão de performance: medir tempo de carregamento inicial do iframe Flutter (<3s); verificar ausência de janking no debounce 300ms; cachear build Flutter com Service Worker ou headers.
  • todo Auditoria de acessibilidade: testar navegação por teclado no inspector e paleta; verificar contraste WCAG AA com ferramenta automatizada (axe-core ou Lighthouse).
  • todo Security Gate 3: revisão de código focada em: (1) Validação de origin no postMessage (F4); (2) JWT — expiração, secret em env, não em código; (3) Tenant isolation — query sempre filtra por projectId do JWT; (4) CORS — origem restrita; (5) Sanitização de input no inspector (XSS via props de widget). Segurança
  • todo Testes E2E (Playwright): fluxo completo — abrir editor → arrastar 3 widgets → editar props → salvar → reabrir → verificar rascunho restaurado.
  • todo Sessão de homologação com o board: validar os 8 itens do DoD (PRD seção 7). Particular atenção ao item 7: "board avalia como visualmente agradável e intuitivo".
  • todo Atualizar docs/manifest.json com status do plano e registrar aprovação do board. Fechar issue BMJ-3.

Critério de saída: Todos os 8 itens do DoD verificados pelo board; Security Gate 3 concluído; testes E2E passam; I1 homologado.

Decisões técnicas de projeto

AspectoDecisãoRazão
Gerenciamento de estado Zustand + middleware immer + histórico Menos boilerplate que Redux; undo/redo nativo via zustand-middleware-history; performático para atualizações frequentes (debounce 300ms).
Drag-and-drop @dnd-kit/core + @dnd-kit/sortable Acessível por padrão; funciona sem eventos HTML5 DnD (compatível com iframe como drop target); suporte a sortable (tree) e free-drag (paleta→canvas).
Seleção via canvas Flutter Overlay React transparente sobre o iframe O iframe Flutter não expõe eventos DOM ao React. O overlay detecta cliques e usa coordenadas + ordem dos blocos para inferir qual foi tocado. Alternativa: postMessage de seleção do Flutter (mais preciso, mais complexo — validar na Fase 7).
Formulário do Inspector Gerador customizado a partir do schema Zod Evita dependência de biblioteca de forms pesada (react-hook-form + zod-resolver seriam opções, mas o gerador customizado dá controle total sobre o visual do inspector).
Build Flutter Asset estático em apps/web/public/preview/ Mesma arquitetura validada no WidgetMill. Build gitignored; rebuild via script. Cache por Service Worker ou HTTP cache.
Hierarquia de blocos (I1) Lista plana (sem aninhamento em I1) A spec I1 usa blocks: Block[] linear. Aninhamento real (Row/Column com filhos) requer recursão no renderer e no editor — adicionado em I2/I3 com a funcionalidade de condições e widget avançado.

Riscos e mitigações

RiscoProbabilidadeMitigação
Seleção via canvas Flutter complexa (overlay vs postMessage) Alta Protótipo técnico na Fase 7 antes de commitar abordagem. Fallback: seleção apenas pela Tree View (US-03 parcialmente coberta).
Performance do iframe Flutter (TTI > 3s) Média Build pré-compilado; Service Worker para cache; medir na Fase 4 antes de integrar ao editor.
UX/visual reprovado no board (critério de aceite de 1ª classe) Média Validação visual iterativa: ao final das Fases 1, 5 e 7, apresentar ao board. Não acumular dívida visual para a Fase 8.
Schemas Zod incompletos para alguns widgets da POC Média Auditoria completa na Fase 2. Widgets com schema incompleto são marcados como "limitado" e exibem inspector simplificado em I1.
Drag-and-drop para iframe (drop target) sem suporte em @dnd-kit Baixa A paleta arrasta para fora do iframe; a inserção é calculada por posição relativa ao canvas — não exige evento de drop dentro do iframe.

Resumo de ações de segurança

Os itens abaixo são requisitos de segurança obrigatórios, não opcionais. O Security Gate 3 (Fase 8) verifica cada um deles antes da homologação.
ItemFaseAção
postMessage origin validationF4Allowlist de origens em build time no sdui_preview; rejeitar silenciosamente origens desconhecidas.
postMessage payload no editorF5Nunca incluir tokens ou segredos no payload; enviar apenas PageFragment.
JWT secret em envF3Segredo nunca em código; expiração configurável; .env nunca commitado.
Multi-tenant isolationF3projectId sempre extraído do JWT, nunca do body; toda query filtra por projectId.
Payload validation (API)F3DTO + schema Zod validam antes de persistir; JSON arbitrário rejeitado.
CORS restritoF3Origem permitida via ALLOWED_ORIGIN env; Helmet para headers de segurança.
XSS via props do inspectorF7Props renderizadas no Flutter (não em DOM HTML) — risco baixo; mas o painel inspector React deve sanitizar strings antes de exibir labels/tooltips.

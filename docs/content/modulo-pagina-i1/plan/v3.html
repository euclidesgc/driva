Plano de Implementação — Módulo Página (I1) v3

Aguardando aprovação do board (grillme)  Plano v3   2026-06-24 · Issue BMJ-16 · Architect

v3 — mudança de stack radical. Este plano substitui o v2 que assumia React (Next.js) + NestJS + Zustand + @dnd-kit. O v3 incorpora as decisões do board (BMJ-16): (1) Flutter Web puro como frontend do editor — sem React, sem iframe, sem postMessage; (2) Arquitetura modular Flutter (get_it + flutter_bloc + go_router + fpdart + equatable), conforme artigo de referência; (3) Backend NestJS diferido — I1 usa InMemoryPageRepository + interfaces; NestJS entra em I2+. O plano v2 é superado em sua totalidade. Ver v2 (superado).
Documentos-base: Spec v3 · PRD v2 · docs/content/padrao-de-projeto-flutter.md (artigo de arquitetura — leitura obrigatória) · docs/WidgetMill/ (referência técnica — renderer a portar) · docs/web-prototipe/ (referência visual primária).
Critério de aceite de 1ª classe (do board). Uma página funcional com visual/UX ruim não passa no I1. O design system e a fidelidade ao web-prototipe são validados em cada fase — não só no final.
⚠ Divergências com a Spec v3 — a reportar ao CEO para realinhamento.
  1. Spec v3, seção 6 "Preview fiel": descreve preview via iframe + postMessage (sdui_preview separado, validação de event.origin). Com Flutter Web puro, o preview é o widget SduiRenderer embutido diretamente no canvas do editor — sem iframe, sem postMessage, sem origin validation. A arquitetura simplifica; a fidelidade é a mesma. Recomendação: atualizar Spec v3 seção 6 para refletir o preview inline Flutter.
  2. Spec v3, seção 7 NFR Segurança: menciona "Auth JWT via env. Multi-tenant via project_id." Com InMemoryPageRepository em I1, authn/authz e isolamento multi-tenant são diferidos para I2+ (quando NestJS entrar). Em I1 não há backend real, logo não há superfície de authn. Recomendação: mover esse NFR para I2 na Spec.
  3. Spec v3, seção 8, decisão 6: "Estado global do editor: Zustand + immer" — é tecnologia React. Com Flutter Web puro, o estado é EditorCubit (flutter_bloc) + histórico de snapshots imutáveis.
  4. Spec v3, seção 8, decisão 7: "DnD: @dnd-kit/core + sortable" — é biblioteca React. Com Flutter Web puro, o DnD usa Draggable/DragTarget nativos do Flutter.

Ação necessária: CEO deve apresentar estas divergências ao board para confirmar que o plano v3 supera as decisões antigas da Spec (ou pedir atualização da Spec antes de iniciar o desenvolvimento).

Arquitetura de Packages

O I1 é construído como 4 packages com fronteiras de responsabilidade claras, seguindo o padrão do artigo de referência (Clean Architecture + barrels + get_it):

driva_domain/
Dart puro. Entidades (PageFragment, Block), contratos (IPageRepository), use cases, Either<Failure,T>, catálogo de widgets.
driva_repository_memory/
Dart puro. InMemoryPageRepository implementa IPageRepository. Descartado quando NestJS entrar em I2+.
sdui_renderer/
Flutter. Renderer SDUI portado da POC WidgetMill. Widget SduiRenderer que renderiza PageFragment recursivamente.
driva_builder/
App Flutter Web. Editor completo: módulos editor, paleta, tree, inspector. Entry point do Driva Builder.
Regra de dependência (estrita): driva_domain não depende de nada (Dart puro). driva_repository_memory depende de driva_domain. sdui_renderer depende de driva_domain. driva_builder depende dos três anteriores. Nunca inverter a seta.

// Recebe EditorCubit via construtor; chama updateBlockProp com debounce 300ms
class InspectorCubit extends Cubit<InspectorState> {
  final EditorCubit editorCubit;
  Timer? _debounce;
  void updateProp(String blockId, String key, Object? value) {
    _debounce?.cancel();
    _debounce = Timer(const Duration(milliseconds: 300), () {
      editorCubit.updateBlockProp(blockId, key, value);
    });
  }
}

Integração sdui_renderer ↔ driva_builder

Com Flutter Web puro, o preview não usa iframe nem postMessage — o SduiRenderer é um widget Flutter ordinário embutido dentro do canvas do editor. Essa simplicidade é a vantagem fundamental sobre a arquitetura React + iframe do plano v2. 

// canvas_panel.dart
class CanvasPanel extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return BlocBuilder<EditorCubit, EditorState>(
      builder: (context, state) {
        if (state is! EditorLoaded) return const SizedBox();
        return Stack(children: [
          PhoneMockup(
            device: state.device,
            child: ClipRect(
              child: SduiRenderer(       // package sdui_renderer
                page: state.page,
                selectedBlockId: state.selectedBlockId,
                onBlockTap: (blockId) => context.read<EditorCubit>().selectBlock(blockId),
              ),
            ),
          ),
          // Overlay de DnD (DragTarget) sobre o mockup
          CanvasDragOverlay(device: state.device),
        ]);
      },
    );
  }
}

O SduiRenderer aceita selectedBlockId e renderiza um outline de seleção sobre o bloco correspondente — sem protocolo de rede, apenas passagem de parâmetro.

DnD canvas: O CanvasDragOverlay é um DragTarget<WidgetDef> empilhado sobre o renderer. Ao aceitar um drop, chama editorCubit.insertBlock(null, 'children', -1, def) para inserir ao final da raiz (cenário A). Containers exibem seu próprio DragTarget filho quando em foco durante drag (cenário B/C).

DI com get_it — setup

// driva_builder/lib/injection.dart
final getIt = GetIt.instance;

void setupLocator() {
  registerEditorModule(getIt);     // repositório, use cases, cubit factory
  registerPaletaModule(getIt);
  registerTreeModule(getIt);
  registerInspectorModule(getIt);
}

// editor_injection.dart
void registerEditorModule(GetIt getIt) {
  // Repositório: lazy singleton (troca InMemory por NestJSImpl em I2+ sem mudar nada acima)
  getIt.registerLazySingleton<IPageRepository>(() => InMemoryPageRepository());

  getIt.registerFactory(() => GetPageUseCase(repository: getIt()));
  getIt.registerFactory(() => SavePageUseCase(repository: getIt()));
  // EditorCubit: factory (um por rota; dispose automático pelo BlocProvider)
  getIt.registerFactory(() => EditorCubit(
    getPage: getIt(),
    savePage: getIt(),
  ));
}

Visão Geral do Plano

O I1 é construído em 7 fases, com as Fases 2, 3 e 4 rodando em paralelo após a Fase 1. A sequência garante: (a) domínio tipado e testado antes da UI; (b) renderer validado antes do canvas do editor; (c) DI configurada antes dos módulos de UI; (d) design system guia o desenvolvimento desde a Fase 1.

F2 driva_domain paralelas F3 sdui_renderer paralelas F4 repo_memory
→ F5 Builder Shell & Canvas → F6 Paleta, DnD & Tree → F7 Inspector, Persistência & Homologação

Security Gate 2 — Avaliação de Superfície Sensível

✓ Avaliação concluída — sem superfície sensível neste incremento.
ÁreaStatus em I1Observação
Autenticação / AutorizaçãoDiferida → I2+I1 usa InMemoryPageRepository sem backend real. Sem authn, sem tokens, sem surface de ataque.
Dados pessoais (LGPD)Não coletadosEditor edita estrutura de página SDUI, sem PII.
PagamentosFora de escopoNão faz parte do produto Driva Builder em nenhum incremento.
Integrações externasNenhuma em I1InMemory não abre socket. Flutter Web roda no browser, sem chamadas a serviços externos.
iframe / postMessageEliminadoPlano v3 usa Flutter Web puro sem iframe. Toda a superfície de origin validation do plano v2 foi eliminada por design.
XSS (props do inspector)Risco baixo — monitorarProps de widget renderizadas pelo SduiRenderer Flutter — não HTML/DOM — sem vetor de XSS clássico. Props de tipo string exibidas via Text widget (imune a injeção de HTML).
Injeção de códigoNão aplicável em I1Sem execução de código de usuário. Props são strings/números/booleanos mapeados a atributos visuais.
Preparação para I2+ (NestJS)A planejarQuando NestJS entrar: authn JWT, multi-tenant isolation, CORS, Helmet, payload limits — Security Gate 2 deve ser re-executado para I2.

Conclusão: O I1 com Flutter Web puro + InMemory tem a menor superfície de ataque possível. Nenhuma tarefa de segurança bloqueia o desenvolvimento do I1. As preocupações de segurança de rede (authn, CORS, tenant isolation) devem compor o Security Gate 2 de I2 quando NestJS for adicionado. O Security Gate 3 (revisão de código pré-homologação) permanece obrigatório — foco em XSS de props, acesso a recursos do browser e integridade do build Flutter Web.

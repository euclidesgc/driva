PRD — Módulo Página (I1) v2

Aguardando aprovação do board  PRD  v2 — layout corrigido   2026-06-22 · Issue BMJ-1

⚠ Correção v2. O PRD v1 descrevia um layout de 3 colunas com "lista de blocos" (Organização) separada, inspirado na POC WidgetMill de 4 colunas. Após revisão do web-prototipe (protótipo oficial do board), o layout correto é: paleta/tree à esquerda · canvas central com phone mockup · propriedades à direita — igual ao FlutterFlow. Os widgets são arrastados diretamente para a tela do celular, não para uma lista. Esta v2 corrige o layout e alinha o PRD ao protótipo oficial.
Resumo Executivo. O I1 entrega o núcleo do Page Builder da plataforma Driva: um editor web (3 painéis) onde times de produto montam páginas SDUI arrastando widgets da paleta diretamente para um canvas central com moldura de celular, ajustam propriedades no painel direito e veem em tempo real um preview Flutter 100% fiel ao app. O resultado salvo é um JSON de página renderizável em tempo de execução. Critério de aceite de 1ª classe: qualidade de UX e design.

1. Problema

Times que mantêm apps Flutter precisam alterar layouts e conteúdo de páginas com frequência. Cada mudança hoje exige ciclo completo de desenvolvimento → PR → stores, levando dias. A POC WidgetMill validou a abordagem SDUI (páginas como JSON, renderizadas sem nova publicação), mas teve design e UX ruins demais para adoção real.

3. Layout do editor — 3 painéis CORRIGIDO em v2

① PAINEL ESQUERDO

Modo Paleta: widgets disponíveis em grid (icon + nome), com busca e agrupamento por categoria.

Modo Tree: hierarquia dos widgets na página atual, selecionáveis e reordenáveis.

Toggle no cabeçalho do painel.

② CANVAS CENTRAL

Phone mockup renderizado pelo Flutter Web (iframe).
Toolbar com zoom (+/−), seletor de device (smartphone/tablet) e dimensões.

③ PAINEL DIREITO

Inspector: quando nenhum widget selecionado, mostra Props da Página (nome, parâmetros). Quando um widget selecionado, mostra suas propriedades em formulário gerado pelo schema Zod.

Topbar: logo Driva · breadcrumb (App / NomeDaPágina) · barra de busca global · botões (código, abrir em nova aba, Preview, Publish). Em I1 o botão Publish aparece mas direciona ao fluxo de publicação (I4) — fica visível mas mostra tooltipo "disponível no I4".

Painéis redimensionáveis via drag nos divisores entre painel esquerdo/canvas e canvas/painel direito.

Por que não existe coluna "Organização" separada? diferença vs v1
Na POC WidgetMill havia 4 colunas, incluindo uma coluna de ordenação de blocos. O protótipo oficial do Driva (e o FlutterFlow que nos inspira) elimina essa coluna: a organização visual é feita diretamente no canvas (arrasto) e pela Tree View no painel esquerdo (toggle). Isso reduz a carga cognitiva e torna a edição mais intuitiva.

4. Histórias de usuário (I1)

US-01 · CORE ATUALIZADA v2

Arrastar widget da paleta para o canvas

Como criador de página, quero arrastar um widget da paleta para a moldura do celular no canvas, para inserir o widget na página sem escrever código.

Critérios de aceite:
  • A paleta exibe os ~14 primitivos Flutter (Container, Row, Column, Text, Image, Icon, Button, Spacer, Divider, Card, Stack, Padding, Expanded, SizedBox) em grid com icon + nome.
  • Arrastar da paleta → soltar no canvas (phone mockup) insere o widget com valores padrão na posição correta.
  • A zona de drop no canvas é visualmente indicada durante o arrasto (highlight de área).
  • Após o drop, o widget aparece imediatamente no preview Flutter e é automaticamente selecionado.
US-02 · CORE ATUALIZADA v2

Reordenar e remover widgets pela Tree View

Como criador de página, quero ver e reorganizar a hierarquia da página na Tree View, para controlar a ordem e o aninhamento dos widgets sem rearranjar manualmente no canvas.

Critérios de aceite:
  • O painel esquerdo tem toggle "Paleta / Tree" no cabeçalho.
  • A Tree View mostra os widgets da página em hierarquia (nó pai ↔ filhos) com ícone e nome.
  • Drag-and-drop dentro da tree reordena ou reaninha widgets; canvas atualiza com debounce ≤ 300 ms.
  • Clicar em nó da tree seleciona o widget (destaca no canvas e abre inspector no painel direito).
  • Cada nó tem ação de remover (botão visível no hover), com confirmação visual.
US-03 · CORE

Selecionar widget no canvas e editar propriedades

Como criador de página, quero clicar num widget no canvas para selecioná-lo e editar suas propriedades no painel direito, para configurar conteúdo e visual de cada widget.

Critérios de aceite:
  • Clicar num widget no canvas o seleciona (outline de seleção visível, label no topo do outline).
  • O painel direito exibe o inspector do widget selecionado com formulário gerado pelo schema Zod.
  • Cada alteração no inspector reflete no canvas com debounce ≤ 300 ms.
  • Clicar fora de qualquer widget no canvas desmarca a seleção e mostra Page Props no painel direito.
US-04 · CORE ATUALIZADA v2

Canvas com zoom e seletor de device

Como criador de página, quero controlar o zoom do canvas e mudar o preset de dispositivo, para verificar como a página aparece em telas de diferentes tamanhos.

Critérios de aceite:
  • Toolbar do canvas tem controles de zoom (+/−) com % exibida (padrão: 90%).
  • Seletor de device com pelo menos 3 presets: Smartphone (ex. 393×852 px), Tablet, Desktop.
  • Trocar device redimensiona a moldura sem recarregar o app Flutter no iframe.
  • A moldura do celular é fiel ao físico (bordas arredondadas, notch).
US-05 · CORE

Salvar e reabrir página como rascunho

Como criador de página, quero salvar o rascunho e reabri-lo depois, para continuar o trabalho em sessões diferentes.

Critérios de aceite:
  • Botão "Salvar rascunho" na topbar persiste o JSON da página no backend (NestJS, project_id-scoped).
  • Status de salvamento ("Salvo" / "Editando…") visível na topbar, discreto mas claro.
  • Ao reabrir o editor, o último rascunho é restaurado automaticamente.
  • O JSON salvo segue a spec definida na Spec do I1 (seção 4).
US-06 · UX — critério de aceite

Editor com qualidade visual de produto comercial

Como usuário do Driva, quero uma interface que me dê prazer em usar, para adotar a ferramenta no meu fluxo de trabalho sem atrito.

Critérios de aceite:
  • Visual fiel ao web-prototipe: fundo cinza-claro no canvas, painéis brancos, accent roxo/azul (#6c4cf1 ou similar), tipografia Public Sans.
  • Design system consistente: tokens de cor, espaçamento, bordas, sombras e estados de interação padronizados.
  • Interações fluidas: arrasto com ghost preview, hover states, transições suaves.
  • Undo/redo (Ctrl+Z / Ctrl+Y) para todas as ações de edição.
  • Navegação por teclado no inspector (acessibilidade WCAG AA).
  • Obrigatório: uma página funcional com visual/UX ruim não passa no aceite.

7. Definição de "pronto" (I1)

  1. ✓ PM consegue montar página com 3+ widgets arrastando da paleta ao canvas sem assistência.
  2. ✓ Preview no canvas é idêntico ao Flutter real (zero divergência visual).
  3. ✓ Tree View mostra hierarquia correta e permite reordenar / remover.
  4. ✓ Troca de device redimensiona canvas sem reload.
  5. ✓ Página salva como rascunho e reabrível em nova sessão.
  6. ✓ Undo/redo funciona para todas as ações de edição.
  7. ✓ Board avalia o editor como visualmente agradável e intuitivo — UI ruim = I1 não concluído.
  8. ✓ Security Gate 3 concluído antes da homologação.

8. Fora do escopo do I1

  • Condições de exibição por widget e filtros por propriedade de usuário → I2.
  • Construtor de widget personalizado com estados e fonte de dados → I3.
  • Workflow rascunho → homologação → produção, papéis, versionamento → I4.
  • Serving de páginas para app de produção (GET /pages/{id}) — preview interno basta no I1.
  • Auth granular por role — I1 usa JWT único por projeto.

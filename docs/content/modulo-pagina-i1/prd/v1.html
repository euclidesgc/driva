PRD — Módulo Página (I1)

Aguardando aprovação do board  PRD   v1 · 2026-06-22 · Issue BMJ-1

Resumo Executivo. O I1 entrega o núcleo do Page Builder da plataforma Driva: um editor web onde times de produto montam páginas SDUI arrastando widgets, organizam a ordem dos blocos, editam propriedades e visualizam o resultado em tempo real num preview Flutter 100% fiel ao app do cliente final — tudo isso sem publicar nova versão do app. O resultado salvo é um JSON de página renderizável em tempo de execução. O critério de aceite de primeira classe é a qualidade de UX e design (o maior problema da POC).
Documentos de referência:
  • Spec do I1 — escopo técnico, modelo de dados, anatomia do editor, decisões do board.
  • Visão Geral — posicionamento, 3 módulos, 4 incrementos e mapa de evolução.
  • web-prototipe (repositório local, docs/web-prototipe/) — referência visual primária do editor de 3 colunas.
  • WidgetMill POC (repositório local, docs/WidgetMill/) — kernel técnico a reaproveitar (sdui_flutter, sdui_preview, spec Zod).

1. Problema

Equipes que mantêm apps Flutter precisam alterar o layout e o conteúdo de páginas do app com frequência. Cada mudança hoje exige um ciclo completo de desenvolvimento Dart → PR → review → build → publicação nas stores, levando dias ou semanas. Não existe forma de iterar rapidamente em páginas de marketing, promoções ou home sem envolver engenharia.

A POC WidgetMill validou a abordagem de Server-Driven UI (SDUI): páginas definidas como JSON, renderizadas em tempo de execução pelo Flutter sem nova publicação. O que faltou na POC foi uma interface de edição com qualidade suficiente para ser adotada por equipes de produto — o design era ruim, a UX confusa, e a ferramenta não gerava confiança.

QA / Homologador Revisor Abrir a página salva e verificar que o preview corresponde à expectativa antes de publicar (I4).

3. Proposta de valor

VELOCIDADE Mudança de layout de dias para minutos — sem PR, sem store, sem engenharia mobile envolvida.
CONFIANÇA Preview que usa o mesmo renderer do app: zero divergência entre editor e produção. O que você vê é o que o usuário vê.
CONTROLE Salvar rascunhos, reabrir e iterar antes de publicar (fluxo completo de publicação chega no I4).

4. Histórias de usuário (I1)

US-01 · CORE

Montar uma página arrastando widgets

Como criador de página, quero arrastar widgets da paleta para a área de trabalho da página, para compor o layout sem escrever código.

Critérios de aceite:
  • A paleta exibe os ~14 primitivos Flutter da POC (Container, Row, Column, Text, Image, Icon, Button, Spacer, Divider, Card, Stack, Padding, Expanded, SizedBox).
  • Arrastar um widget da paleta para a lista de blocos insere o widget com valores padrão.
  • O widget aparece imediatamente no preview Flutter.
  • A zona de drop é visualmente clara durante o arrasto (outline, cor de destaque).
US-02 · CORE

Reordenar e remover blocos da página

Como criador de página, quero reordenar e remover os blocos da página, para controlar a estrutura final sem precisar recriar tudo.

Critérios de aceite:
  • Blocos podem ser reordenados por arrastar-e-soltar dentro da coluna de organização.
  • Cada bloco tem um botão/ação de remover, com confirmação visual (hover/estado).
  • Ao remover ou reordenar, o preview atualiza com debounce ≤ 300 ms.
US-03 · CORE

Editar propriedades de um bloco

Como criador de página, quero selecionar um bloco e editar suas propriedades num inspector lateral, para configurar o conteúdo e o visual de cada widget.

Critérios de aceite:
  • Selecionar um bloco na coluna de organização abre o inspector na coluna de propriedades.
  • O inspector exibe um formulário gerado dinamicamente com base no schema do widget (herdado da spec Zod da POC).
  • Cada alteração no inspector reflete no preview com debounce.
  • O bloco selecionado é visualmente destacado na coluna de organização.
US-04 · CORE

Ver o preview fiel em moldura de dispositivo

Como criador de página, quero ver a página renderizada num preview Flutter com opção de moldura de dispositivo (iPhone / Android / tablet), para garantir que o layout ficará correto antes de salvar.

Critérios de aceite:
  • O preview usa o mesmo sdui_flutter da POC (zero divergência de renderização).
  • Há ao menos 3 presets de dispositivo selecionáveis (ex.: iPhone 14, Pixel 7, iPad Air).
  • A troca de dispositivo redimensiona a moldura sem recarregar o Flutter app.
  • O preview responde a alterações de blocos/propriedades com debounce ≤ 300 ms via postMessage.
US-05 · CORE

Salvar e reabrir uma página como rascunho

Como criador de página, quero salvar a página como rascunho e reabri-la depois, para continuar o trabalho em sessões diferentes.

Critérios de aceite:
  • Botão "Salvar rascunho" na topbar salva a página no backend (NestJS) com project_id e identificador único.
  • A topbar exibe o status de "Salvo" / "Editando…" de forma não intrusiva.
  • Ao reabrir a sessão do editor, a última versão do rascunho é restaurada automaticamente.
  • Salvar gera um JSON conforme a spec definida na Spec do I1 (seção 4).
US-06 · UX

Navegar pelo editor com qualidade de produto comercial

Como usuário do Driva, quero uma interface moderna, limpa e responsiva, para confiar na ferramenta e adotá-la como parte do meu fluxo de trabalho.

Critérios de aceite:
  • O layout de 3 colunas segue a referência visual do web-prototipe (paleta · organização · properties+preview).
  • Painéis são redimensionáveis pelo usuário.
  • Design system consistente: tipografia, espaçamento, cores e estados de interação (hover, focus, active, disabled) padronizados.
  • Suporte a undo/redo (Ctrl+Z / Ctrl+Y) nas ações de arrastar, reordenar, remover e editar propriedades.
  • Navegação por teclado nos controles do inspector (acessibilidade).
  • Obrigatório: uma página funcional porém com visual/UX ruim não passa no aceite.

⚠ Design system pendente. O board está produzindo o design system no Claude Design (localização a confirmar). Até sua disponibilização, a referência visual primária é o protótipo docs/web-prototipe/ (layout de 3 colunas, paleta lateral, preview à direita) e o padrão de qualidade visual do FlutterFlow (nossa referência de UX de mercado).   • Na fase de UX, construir wireframes de alta fidelidade a partir do web-prototipe.
  • Ao receber o design system do board, incorporá-lo antes da implementação final de UI.

8. Definição de "pronto" (I1)

O I1 está concluído quando todos os itens abaixo forem verdadeiros:

  1. ✓ Um PM consegue montar uma página com pelo menos 3 widgets distintos da paleta sem assistência.
  2. ✓ O preview exibe a página de forma idêntica ao app Flutter real (zero divergência visual).
  3. ✓ A página é salva como rascunho e pode ser reaberta e continuada em nova sessão.
  4. ✓ Undo/redo funciona para todas as ações de edição.
  5. ✓ O editor foi avaliado pelo board como visualmente agradável e intuitivo — UI ruim = I1 não concluído.
  6. ✓ Security Gate 3 concluído: código revisado pelo agente de Security antes da homologação.

9. Fora do escopo do I1

  • Condições de exibição por bloco e filtros por propriedade de usuário → I2.
  • Construtor de widget personalizado com estados (loading/success/empty/error) e fonte de dados → I3.
  • Workflow rascunho → homologação → produção, papéis, versionamento e agendamento → I4.
  • Serving de páginas para o app de produção (GET /pages/{id}) — em I1 o preview interno basta.
  • Autenticação/autorização granular por role — I1 usa auth simples (API key / JWT único).

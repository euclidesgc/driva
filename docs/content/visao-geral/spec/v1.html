Driva — Visão Geral & Recorte em Incrementos

Decisões de framing aprovadas   Versão de trabalho · 2026-06-21 · Issue BMJ-2

Status deste documento. Esta é a visão geral do produto e a proposta de como fatiá-lo em incrementos curtos. Ela não é a especificação detalhada de nenhum módulo — essa virá em seguida, para o incremento que o board escolher como primeiro. Antes de detalhar, há 4 decisões pendentes que dependem do board (seção final).

1. O que é o Driva

O Driva é uma plataforma de Server-Driven UI (SDUI): uma aplicação web onde times montam páginas e widgets que são renderizados no lado do app Flutter do cliente final, sem precisar publicar uma nova versão do app. O produto da plataforma é dado (um spec em JSON), não código Dart — o app baixa esse spec e o renderiza com um renderer proprietário.

A ferramenta é composta por módulos. Três foram descritos pelo board (seção 3).

2. Aprendizados da POC WidgetMill

A POC WidgetMill (em /home/euclidesgc/Development/WidgetMill) validou a parte mais arriscada da arquitetura: o contrato spec ↔ renderer ↔ preview fiel. Vale herdar o que ela de-riscou e superar o que ela deixou em aberto.

O que vale aproveitar

  • Spec como kernel único — uma só fonte de verdade (Zod) da qual derivam validação, tipos, forms do Inspector e fixtures. Congelar o spec é a 1ª tarefa de código.
  • Preview 100% fiel — app Flutter Web (sdui_preview) embarcado em <iframe>, comunicação por postMessage, molduras de dispositivo. Zero divergência React↔Flutter.
  • Renderer proprietário (sdui_flutter) por registry type → builder + dispatcher de ações (estilo Stac), reusado tanto no preview quanto no app de produção.
  • Catálogo de primitivos espelhando widgets do Flutter; eventos → lista de ações como dado; binding {{prop}}.
  • Governança — multi-tenant por project_id, papéis (Reader/Creator/Reviewer/Publisher/Admin), workflow como máquina de estados, draft + versionamento imutável.
  • Stack — Next.js (editor web), NestJS (backend de domínio), Flutter Web (preview), Postgres/JSONB, Puck (drag-and-drop).

O que a POC deixou em aberto e o Driva precisa superar

  • Não tinha o Módulo de Página (montagem de página + condições de exibição + filtros por widget + simulação de propriedades de usuário). Era só o construtor de widgets.
  • Widgets só primitivos — sem componentes complexos com estados (loading/success/error/idle/empty) nem fonte de dados com parâmetros (o que o board agora pede, à la FlutterFlow).
  • Sem agendamento de publicação e sem fluxo rascunho → homologação → produção aplicado a páginas.
  • Outros problemas concretos da POC que motivaram recomeçar: a confirmar com o board (ver Decisão 1).

3. Os módulos do produto

Módulo A Construção de Página (Page Builder / SDUI)

O coração do produto. Layout em colunas, no espírito descrito pelo board:

ColunaFunção
Esquerda — PaletaLista de widgets disponíveis para arrastar e soltar.
Centro — OrganizaçãoOs widgets que comporão a página, em ordem; cada um com seu filtro de exibição (exibe ou não conforme configuração).
PropriedadesQuando um widget está selecionado, suas propriedades editáveis.
Direita — PreviewComo a página aparece no app, com simulação de propriedades de usuário para ver quais widgets seriam exibidos.

A página também tem condições de exibição próprias (a página inteira aparece ou não no SDUI).

Módulo B Construção de Widget (estilo FlutterFlow)

  • Compor componentes complexos a partir dos widgets padrão do Flutter (evolução do construtor da POC).
  • Estados do componente: loading, success, error, idle, empty — cada um com seu próprio layout.
  • Fonte de dados para o componente, com passagem de parâmetros (o componente busca dados e troca de estado conforme o resultado).

Módulo C Fluxo de Trabalho, Versionamento & Agendamento

  • Publicação rascunho → homologação → produção, com papéis/usuários responsáveis por cada etapa.
  • Agendamento de publicação — de widgets e de páginas.
  • Versionamento de widgets e de páginas (versão publicada imutável).

4. Proposta de recorte em incrementos

Regra do Driva: ciclos curtos. Não detalhamos nem construímos os 3 módulos de uma vez. A proposta abaixo é a ordem sugerida; o board decide (Decisão 2).

Inc.Escopo (valor entregue)Por quê nesta ordem
I1 Página Montar uma página com um conjunto pequeno de widgets prontos, ordená-los, editar propriedades e ver o preview fiel ponta a ponta. Sem filtros complexos ainda. Entrega o produto-essência (construção de página) e reusa o renderer/preview já de-riscado pela POC.
I2 Página Condições de exibição da página + filtros por widget + simulação de propriedades de usuário no preview. Diferencial de SDUI segmentado; depende da página existir (I1).
I3 Widget Construtor de widget evoluído: estados (loading/success/error/idle/empty) + fonte de dados com parâmetros. Enriquece o catálogo que as páginas consomem; é o módulo mais profundo.
I4 Fluxo Workflow rascunho→homologação→produção com papéis + versionamento + agendamento de publicação (páginas e widgets). Governança de publicação faz mais sentido quando já há páginas e widgets para publicar.

Cada incremento terá sua própria spec detalhada + PRD + plano em fases, um de cada vez. Os demais ficam no backlog.

5. Arquitetura herdada (a confirmar no detalhamento)

Entendimento atual da stack, herdado da POC: Next.js (editor web e telas de gestão), NestJS (backend de domínio: RBAC, workflow, versionamento), Flutter Web (app de preview fiel), Postgres/JSONB, Puck (canvas drag-and-drop). O app cliente final é Flutter nativo, que consome o spec e o renderiza com o sdui_flutter. A frase do board "já temos vários tipos de sistema: Next, Nest e Flutter Web" é entendida como essa stack — a confirmar (Decisão 3).

6. Decisões do board

✓ Resolvidas em 2026-06-21 (BMJ-2):
  1. Problema-chave da POC a superar: o design/UX da página estava ruim. O board quer uma ferramenta com UX melhor e design mais atrativo e intuitivo → UX/visual é requisito de primeira classe no Driva.
  2. Primeiro incremento: I1 — Módulo Página (montagem + preview fiel). Spec detalhada: modulo-pagina.html.
  3. Stack: "Next, Nest e Flutter Web" = a stack da plataforma (Next = editor, Nest = backend, Flutter Web = preview). App cliente final é Flutter nativo.
  4. Ponto de partida: reaproveitar o kernel da POC (spec + sdui_flutter + sdui_preview) e reconstruir editor/backend.

Como as decisões abaixo foram tomadas conforme as recomendações, mantemos o registro original para histórico.

  1. Quais problemas concretos da POC WidgetMill precisamos superar? O board citou que a POC "trouxe alguns problemas", mas não os listou. Saber quais (ex.: performance do preview, tradução Puck↔spec frágil, modelo de dados, DX, arquitetura do backend, etc.) muda diretamente o desenho do Driva. (resposta aberta)

  2. Qual incremento entregar primeiro?

    ▸ Recomendado: I1 — Módulo Página (montagem + preview fiel), pois é a essência do produto e reusa o renderer já provado.
    ○ I3 — Módulo Widget evoluído (estados + fonte de dados) primeiro, para enriquecer o catálogo antes das páginas.
    ○ I4 — Fluxo/versionamento/agendamento primeiro (governança antes da construção).

  3. Confirmação da stack / alvos de renderização.

    ▸ Recomendado: "Next, Nest e Flutter Web" = a stack da plataforma (editor + backend + preview); app cliente final é Flutter nativo.
    ○ São 3 alvos de saída: a mesma página renderiza em Next, em Nest e em Flutter Web (produto multiplataforma de saída).
    ○ Outro — o board esclarece.

  4. Ponto de partida do código.

    ▸ Recomendado: Reaproveitar o kernel da POC (spec + sdui_flutter + sdui_preview) no repo Driva e reconstruir editor/backend conforme os aprendizados.
    ○ Greenfield total no repo Driva, usando a POC apenas como referência conceitual.
    ○ Migrar a POC inteira para o repo Driva e evoluir incrementalmente.
